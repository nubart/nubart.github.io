1. INTRODUCTION

Nubart provides access codes to its audio guides that are unique and non-transferable thanks to a proprietary system.

The API is a REST-oriented HTTP web service and uses built-in HTTP features like HTTP verbs and HTTP error codes for error handling, which makes it easily accessible for off-the-shelf HTTP clients.

All payload data is transmitted in valid JSON format.

2. AUTHENTICATION

Every call to the API has to be made over a secure HTTPS connection and needs the authentication of the client. You can create an access key in your customer area which serves as client identifier and authentication to the API. The key is passed in the HTTP Authorization header as Bearer authentication:

Authorization: Bearer <your_access_key>

IMPORTANT: The access key has to be kept in a safe place and never be transmitted to third parties or over an insecure connection. We recommend replacing the key regularly or every time the key might have been exposed.

3. HTTP RESPONSE CODES

HTTP defines a range of meaningful status codes, some of which will be returned from the API:

Code Meaning
200 OKResponse to a successful request
400 Bad RequestThe request is malformed, e.g. if the body does not parse
401 UnauthorizedWhen no or invalid authentication details are provided
403 ForbiddenWhen authentication succeeded but the authenticated user doesn't have access to the resource
404 Not FoundWhen a non-existent resource is requested
405 Method Not AllowedWhen an HTTP method is being requested that isn't allowed for the authenticated user
410 GoneIndicates that the resource at this endpoint is no longer available
415 Unsupported Media TypeIf an incorrect content type was provided as part of the request
422 Unprocessable EntityUsed for validation errors
429 Too Many RequestsWhen a request is rejected due to rate limiting

Error responses can have the following JSON body:

{
  "error": "invalid request",
  "details": "<error description>"
}

4. ENDPOINTS

The API's root URL is:

https://www.nub.art/customer/api/lwac/v2

4.1 Retrieve the next token

Retrieves the next token along with its series information.

POST/module/<module ID>/nexttoken

The "module ID" is an ID string of the module for which you want to get the next available token and will be provided by Nubart.

The retrieved token is pending and not usable until the "activate" endpoint is called (see section 4.2 below).

This endpoint will never return the same token twice.

Response

Name Type Description
idstringThe ID of the retrieved token
tokenstringThe secure token
urlstringThe URL that should be used to generate the QR code for the end-user
serialnostringThe serial number with the format <number of the token>/<size of the series>-<number of the series>. This is just additional information and corresponds to the serial number on our printed cards.

Example

{
  "id": "66d87e2ca85b3d6753ae515c",
  "token": "880ba6849f557a",
  "url": "https://www.nub.art/qr/880ba6849f557a",
  "serialno": "18/1000-5"
}

If no token is available, the endpoint returns 404 Not Found along with the following body:

{
  "error": "invalid request",
  "details": "no token available"
}

4.2 Activate token

Activates a token that was previously retrieved via nexttoken.

POST/module/<module ID>/activatetoken/<ID>

The ID is the string that was returned by nexttoken in the id field.

This endpoint is idempotent and can be retried in case of failures.

The response is an empty JSON object; the call can be considered successful if the returned status code is HTTP 200.

5. REQUEST LIMITS

The call rate per access key is limited by a token bucket with a default size of 100 and a rate of 1/s.

If this limit is exceeded, the endpoints will return an HTTP 429 response.

TIMEOUTS & RETRIES: Clients should configure a request timeout of at least 20 seconds. Response times may occasionally exceed typical latency (for example, during deployments). On a timeout or a 5xx response, retry the request using exponential backoff before treating the call as failed.
