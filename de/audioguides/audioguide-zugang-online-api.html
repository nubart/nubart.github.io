Verkauf von Nubart GUIDE-Zugängen über Ihren Online-Shop – Leitfaden zur Umsetzung

Auf dieser Seite erfahren Sie, was Ihr Museum wissen muss, bevor Sie Ihren Webentwickler oder Ihren Ticketanbieter damit beauftragen, Ihren Online-Shop mit Nubart GUIDE zu verbinden. Erläutert wird, wie die Integration funktioniert, was Ihr Entwickler benötigt, wie Sie Ihre API-Zugangsdaten sicher weitergeben können und ein Implementierungsdetail, das Besuchern hilft, spätere Probleme zu vermeiden.

Für Ihren Entwickler: Alle technischen Details (Authentifizierung, Endpunkte, Anfrage- und Antwortformat) finden Sie in der Nubart LWAC-API-Dokumentation. Leiten Sie diesen Link direkt an die Person weiter, die die Integration umsetzt.
  Nubart LWAC-API-Dokumentation öffnen

Bevor Sie beginnen

Bevor Sie Ihren Entwickler bitten, die API zu implementieren, stellen Sie bitte sicher, dass Sie Folgendes haben:

  •   Ein Administratorkonto im Nubart-Kundenportal
  •   Festgelegt, welches Modul online verkauft werden soll
  •   Entschieden, wer den API-Schlüssel abrufen wird (siehe unten)
  •   Der Link zur technischen Dokumentation oben, bereit zur Weitergabe an Ihren Entwickler
Besucher kauft ein Ticket
Der Online-Shop fordert einen Zugangscode an
Die Nubart-API gibt den Code zurück
Das Ticket enthält einen QR-Code und einen Direktlink
Der Besucher öffnet den Leitfaden

Was die API tatsächlich liefert

Die API erstellt keine separate Version von Nubart GUIDE. Sie stellt lediglich denselben sicheren Zugangscode digital bereit, anstatt ihn auf eine physische Karte zu drucken. Anstatt einem Besucher am Schalter eine Karte auszuhändigen, fordert Ihr Online-Shop zum Zeitpunkt des Kaufs einen Code bei Nubart an und stellt ihn dem Besucher als Teil seines Tickets zur Verfügung.

Alles, was die Kartenversion auszeichnet – Nichtübertragbarkeit, kein Login, keine App – wird identisch auf die digitale Version übertragen.

So erhalten Sie Ihren API-Schlüssel

Um dies in Ihrem Online-Shop zu implementieren, benötigt Ihr Webmaster Ihren API-Schlüssel. Aus Sicherheitsgründen haben wir als Nubart-Mitarbeiter keinen Zugriff auf diesen Schlüssel. Nur Benutzer mit der Rolle „Admin“ können Ihren API-Schlüssel abrufen. In der Praxis bedeutet dies, dass es einen zusätzlichen Schritt erfordert, ihn Ihrem Webmaster zu übergeben, bevor mit der Implementierung begonnen werden kann.

Option 1: Gewähren Sie Ihrem Webmaster vorübergehend Administratorrechte

Der einfachste Weg besteht darin, Ihren Webmaster in Ihr Nubart-Kundenportal einzuladen, wo er den Schlüssel selbst abrufen kann. Dies können Sie unter dem Reiter „Mitarbeiter“ tun. Der Nachteil dabei ist, dass damit vollständige Administratorrechte verbunden sind, nicht nur der Schlüssel – Ihr Webmaster könnte dann auch beispielsweise Ihre Nutzungsstatistiken für Audioguides einsehen. Wenn Sie dies nicht unbefristet zulassen möchten, können Sie zum Reiter „Mitarbeiter“ zurückkehren und ihm den Zugriff sperren, sobald die Implementierung abgeschlossen ist.

Option 2: Ein Administrator ruft den Schlüssel ab und gibt ihn sicher weiter

Wenn Sie den Portalzugang lieber gar nicht erweitern möchten, kann sich einer der „Admin“-Benutzer Ihres Kontos anmelden, den Schlüssel kopieren und ihn stattdessen über einen sicheren Kanal an den Webmaster senden.

Einige Möglichkeiten zur Weitergabe, grob geordnet von der einfachsten bis zur aufwendigsten:

  • Fügen Sie ihn als gemeinsamen Eintrag in den Passwortmanager Ihres Teams ein und gewähren Sie dem Webmaster Zugriff auf diesen Eintrag.
  • Teilen Sie den Schlüssel in zwei Hälften und senden Sie jede Hälfte über einen anderen Kanal – die erste Hälfte beispielsweise per E-Mail, die zweite per Messenger.
  • Lesen Sie den Schlüssel während eines Telefonats laut vor und lassen Sie den Webmaster ihn direkt in die Umgebungsvariablen des Shops eingeben, sodass er während der Übertragung überhaupt nicht schriftlich festgehalten wird.

Unabhängig davon, welche Methode Sie verwenden, ist das Ziel dasselbe: Vermeiden Sie es, den vollständigen Schlüssel als Klartext in einer einzigen E-Mail oder Chat-Nachricht zu versenden. Die Aufteilung auf verschiedene Kanäle oder über einen bestimmten Zeitraum hinweg ist es, was ihn tatsächlich schützt.

Das eine wichtige Detail bei der Umsetzung: Zeigen Sie stets sowohl den Link als auch den QR-Code an

Wenn ein Besucher ein Ticket online kauft, können Sie nicht im Voraus wissen, welches Gerät er zum Öffnen des Audioguides verwenden wird. Einige werden den Link direkt nach dem Kauf zu Hause auf ihrem Laptop anklicken. Andere werden warten und einen QR-Code mit ihrem Smartphone am Museumseingang scannen.

Wir empfehlen dringend, dass jedes Ticket sowohl den direkten Link als auch den dazugehörigen QR-Code – niemals nur das eine oder das andere. Dies gibt den Besuchern die volle Freiheit, später zu entscheiden, ob sie sofort auf ihrem Smartphone auf den Audioguide zugreifen oder warten möchten, bis sie im Museum angekommen sind. Dies ist bei unserer API Standard; jeder Webentwickler kann dies direkt in Ihre Ticketvorlage implementieren.

Wir empfehlen außerdem, in der Nähe des Links bzw. des QR-Codes einen kurzen Hinweis hinzuzufügen:

„Tipp: Öffnen Sie diesen Link auf dem Smartphone, das Sie während Ihres Besuchs tatsächlich nutzen werden.“

Dieser Text muss von Ihrem Team in Ihre eigene Ticketvorlage eingefügt werden – Nubart fügt ihn nicht automatisch ein.

Was passiert, wenn ein Besucher den Link auf zwei verschiedenen Geräten öffnet

Wenn ein Besucher den Link zu Hause auf einem PC öffnet und später denselben Link auf seinem Smartphone öffnet, wird der Besucher lediglich nach einer E-Mail-Adresse gefragt, damit ein neuer Zugangs-Link an das richtige Gerät gesendet werden kann. Der Besucher erhält weiterhin vollen Zugriff – es geht nichts verloren –, aber es entsteht ein zusätzlicher Schritt.

Sobald ein Besucher auf diese Weise eine E-Mail-Adresse angegeben hat, kann diese nicht mehr geändert werden. Somit verhindert derselbe Mechanismus, der den Zugriff auf einem neuen Gerät wiederherstellt, auch, dass der Code frei an andere Personen weitergegeben wird.

Hinweis: Einige E-Mail-Dienste scannen Links aus Sicherheitsgründen automatisch. Unser System erkennt dies, sodass es Ihre Besucher nicht versehentlich daran hindert, ihren Leitfaden zu öffnen.

Die Berichterstellung bleibt unverändert

Der Zugriff über die API ändert nichts an der Darstellung Ihrer Statistiken. In Ihrem Kundenbereich erscheint ein neuer Eintrag mit dem Namen Ihres Museums und dem Zusatz „(API)“ – zum Beispiel „[Name des Museums] (API)“ –, der angibt, wie viele Zugriffe über die API erfolgten.

In den Modulstatistiken werden beide Besuchergruppen zusammengefasst: Diejenigen, die eine physische Karte verwenden, und diejenigen, die über die API zugreifen, werden zusammengeführt, sodass Sie einen vollständigen Überblick über die Nutzung der Audioguides erhalten.

Sind hierfür Änderungen am Audioguide erforderlich?

Nein. Der Audioguide selbst bleibt genau derselbe. Es ändert sich lediglich die Art und Weise, wie Besucher ihren Zugangscode erhalten.
